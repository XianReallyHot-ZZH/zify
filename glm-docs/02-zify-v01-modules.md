# Zify 一期模块定义

> 约束：一个人开发，面向团队内部 20-50 人使用，本地部署。
> 目标：快速验证产品核心链路和功能效果。
> 注：图中每个逻辑模块对应一个独立的 Maven 子模块（`zify-{module}`），由 `zify-app` 启动模块聚合。

## 1. Agent 管理

### 定位

产品的核心配置入口。用户在此创建和编辑 Agent，配置其人设、模型、工具、知识库等所有能力。

### 功能范围

- Agent CRUD：创建、编辑、复制、删除 Agent
- 基础配置项：
  - **名称与描述**：Agent 的名称和用途说明
  - **System Prompt**：定义 Agent 的角色、行为规则、回答风格，支持 `{{variable}}` 变量注入（如当前日期等）
  - **模型选择**：从已配置的模型中选择该 Agent 使用的 LLM
  - **工具绑定**：从统一工具系统中选择该 Agent 可使用的工具
  - **知识库绑定**：从知识库中选择该 Agent 可检索的知识库
  - **Agent 类型**：创建时选择，决定 Agent 的执行模式
    - **ReAct Agent**：LLM 自主决策调用工具，适合开放式问答、灵活探索场景
    - **Workflow Agent**：绑定一个工作流，执行固定的节点链路，适合流程自动化、定时任务、被外部系统调用等确定性场景
  - 创建 Workflow Agent 时，需要从已创建的工作流中选择一个进行绑定
- Agent 列表：展示所有已创建的 Agent，支持按名称搜索

### 边界（不做）

- 不做 Agent 花名册（Roster）系统，一期 Agent 配置直接绑定在实例上
- 不做 Function Calling 策略，ReAct 已覆盖该场景（循环一轮即结束）
- 不做多套 Prompt 模板切换和 Prompt 版本管理
- 上下文管理（对话历史超窗）：**做** 基础尾部截断兜底 + 摘要压缩（running summary，活窗口 + token 预算触发）；**不做** 相关性裁剪、多级摘要、上下文再排序等更复杂策略
- 不做 Completion App（单轮文本生成）、Advanced Chat、Pipeline App 等 Dify 的其他应用类型

## 2. 模型管理

### 定位

管理 LLM 模型供应商的连接配置，为 Agent 和工作流提供模型调用能力。

### 功能范围

- Provider 配置：填写 API Key、Base URL、模型名称，保存后即可使用
- 模型类型：LLM（对话/生成）、Embedding（向量化）、Rerank（重排序，二期预留）
- 模型测试：配置完成后可发送测试请求验证连通性

### 边界（不做）

- 不做负载均衡（多 Provider 分发）
- 不做多租户 Provider 隔离
- 不做模型用量计费和额度管控
- 不做 TTS / STT 模型类型

### 支持的供应商（一期）

- OpenAI（含兼容 OpenAI 协议的服务，如 Azure OpenAI、DeepSeek、本地 Ollama 等）
- Anthropic
- 其他通过 OpenAI 兼容协议接入的服务

## 3. 统一工具系统

### 定位

为 Agent 和工作流提供可调用的外部能力。所有工具统一抽象为一个 Tool 接口（`name`、`description`、`parameters`、`execute()`），调用方不需要知道背后是哪种类型的工具。

### 功能范围

三种工具实现，共享统一接口：

| 工具类型 | 说明 |
|----------|------|
| **MCP 工具** | 连接外部 MCP Server，自动发现并注册其提供的工具。一期只做 MCP Client |
| **HTTP 工具** | 用户通过 OpenAPI Schema 或手动配置 URL/Header/Body，定义一个 REST API 调用工具 |
| **Workflow-as-Tool** | 将已发布的工作流注册为工具，Agent 或其他工作流可调用。执行时启动一次完整的工作流运行 |

- 工具管理页面：列表查看、启用/禁用、配置编辑
- 工具调用日志：记录每次调用的输入输出，用于调试

### 边界（不做）

- 不做内置工具库（Dify 的 Google Search、DALL-E 等预置工具）
- 不做插件工具系统
- 不做 MCP Server（只做 Client）
- 不做工具 OAuth 认证

### 合并说明

HTTP 工具和工作流中的 HTTP Request 节点是不同层面的东西：
- **HTTP 工具**：在工具管理页面注册一次，Agent 的 ReAct 循环中可被 LLM 自主选择调用
- **HTTP Request 节点**：工作流画布上的一个节点，用户手动编排时使用

## 4. 知识库 RAG

### 定位

让 Agent 和工作流能够检索团队内部的文档知识，提供准确、有据可依的回答。

### 功能范围

- 知识库 CRUD：创建知识库、上传文档、管理文档（启用/禁用/删除）
- 文档解析：支持 PDF、Word（DOCX）、TXT、Markdown 四种格式
- 文本分块：段落级分块，支持用户自定义分块大小和重叠长度
- 向量化存储：使用 PGVector（复用 PostgreSQL），存储文档 chunk 的 Embedding 向量
- 检索：向量相似度检索，返回 Top-K 结果
- 命中测试：知识库详情页内嵌测试 Tab，输入查询文本，返回命中的 chunk 列表和相似度分数，用于调试检索质量

### 边界（不做）

- 不做关键词检索和混合检索（一期只做向量检索）
- 不做 Reranking（检索后重排序）
- 不做 RAG Pipeline（可视化 RAG 编排管道）
- 不做摘要索引（自动生成文档摘要）
- 不做多种分块策略（Q&A 分割、父子分块）
- 不做 Notion 导入、网页抓取、Firecrawl、Watercrawl 等在线数据源
- 不做多种向量数据库（只用 PGVector）

## 5. 简版工作流引擎

### 定位

让用户通过可视化画布编排确定性的多步骤任务流程。与 Agent 的 ReAct 自主决策互补——工作流适合流程固定、步骤明确的场景。

### 功能范围

- 可视化画布：拖拽式节点编排，节点间连线定义执行顺序
- 节点执行引擎：按拓扑顺序依次执行节点，支持条件分支和并行
- 变量系统：节点间通过变量传递数据
- 运行日志：记录每次工作流执行的节点状态、输入输出、耗时

### 一期节点清单（9 个）

| 节点 | 说明 |
|------|------|
| **Start** | 工作流入口，定义输入变量 |
| **End** | 工作流出口，定义输出内容 |
| **LLM** | 调用大语言模型，支持配置 Prompt 和模型选择 |
| **If-Else** | 条件分支，根据条件走向不同路径 |
| **HTTP Request** | 发起 HTTP API 调用，支持配置 URL、Method、Header、Body |
| **Knowledge Retrieval** | 查询知识库，返回相关文档片段 |
| **Code** | 执行用户自定义 Python/JavaScript 代码片段 |
| **Tool** | 调用统一工具系统中已注册的工具（MCP / HTTP / Workflow-as-Tool） |
| **Answer** | 将中间结果输出给用户（对话场景中流式返回） |

### 边界（不做）

- 不做迭代（Iteration）、循环（Loop）节点
- 不做模板转换（Template Transform）、变量赋值（Variable Assigner）、变量聚合（Variable Aggregator）
- 不做问题分类器（Question Classifier）、参数提取器（Parameter Extractor）
- 不做文档提取器（Document Extractor）、列表操作（List Operator）
- 不做数据源节点（Data Source）
- 不做 Human-in-the-Loop（人工审批节点）
- 不做画布实时协作和评论批注
- 不做 DSL 版本管理（导入导出、版本历史、回滚）
- 不做 Agent 子工作流节点

## 6. 触发器

### 定位

让工作流可以自动执行，不需要用户手动触发。底层共享触发器管理服务，只区分触发方式。

### 功能范围

| 触发方式 | 说明 |
|----------|------|
| **Webhook** | 暴露一个唯一的 HTTP 端点，外部系统发送请求时自动触发工作流运行 |
| **定时调度** | Cron 表达式配置，按固定周期自动执行工作流 |

- 触发器管理：在工作流配置中绑定触发器，查看触发记录和运行日志
- 触发器日志：记录每次触发的时间、输入参数、执行状态

### 边界（不做）

- 不做插件触发（Plugin Trigger）
- 不做触发器的复杂条件过滤（如只处理特定 payload 的 Webhook）

---

## 7. 对话管理

### 定位

管理用户与 Agent 的对话记录，支持多轮对话的持续性。

### 功能范围

- 会话列表：按时间倒序展示所有会话
- 会话内消息流：展示完整的多轮对话历史（用户消息、AI 回复、工具调用过程）
- 新建会话：开始一段新的对话
- 删除会话：删除对话及其所有消息
- 继续对话：在已有会话中继续发送消息

### 边界（不做）

- 不做消息反馈（点赞/点踩）
- 不做消息收藏/书签
- 不做标注回复（Annotation）
- 不做会话变量（Conversation Variables）
- 不做语音消息（TTS / STT）

## 一期不做但明确规划的模块

| 模块 | 规划 | 理由 |
|------|------|------|
| 用户认证与权限 | 二期首批 | 一期不设权限，但预留用户标识字段，方便二期加认证时数据迁移 |
| 迭代/循环节点 | 二期 | 配合工作流引擎的深度使用场景 |
| Reranking | 二期 | 知识库检索质量不够时补上 |
| 关键词/混合检索 | 二期 | 向量检索不够用时扩展 |
| 消息反馈与收藏 | 二期 | 用户反馈后按需加 |
| Ops Trace（Langfuse） | 二期 | Agent 调试和可观测性需求 |
| 多种向量数据库 | 按需 | PGVector 够用就不做 |

## 一期明确不做的模块

| 模块 | 理由 |
|------|------|
| 计费系统 | 内部使用，不存在套餐概念 |
| 应用市场 / Explore | 50 人不需要应用发现 |
| 插件系统 | 太重，MCP 已覆盖扩展需求 |
| 内容审核 | 内部团队不需要 |
| 嵌入式聊天组件 | 内部使用直接访问 Web |
| 多租户 / 工作区 | 单团队使用 |
| 实时协作 / 评论 | 50 人不会同时编辑 |
| 企业功能（SSO/合规/遥测） | 内部部署不需要 |
| 自定义品牌 | 内部不需要白标 |
| SDK | 不需要官方 SDK |
| Agent 花名册 | 一期 Agent 直接绑定配置 |
