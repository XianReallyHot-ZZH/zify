# Zify 性能瓶颈预判

> 基于当前部署架构（单 Pod + 50 人使用），按严重程度排序。
> 严重程度 = 发生概率 × 影响范围。

---

## 瓶颈清单

### 1. 外部 LLM API 响应慢且有限流 ⚠️ 最严重

**现象**：用户发消息后长时间无响应，或直接报错 "429 Too Many Requests"。

**触发条件**：
- 10+ 人同时对同一 Provider（如 OpenAI）发起对话，触发 Provider 侧限流（OpenAI 默认 RPM 限制较低）
- 单次 LLM 调用 5-30 秒，ReAct 循环可能连续调用 3-5 次，一个用户的问题就要 15-150 秒
- Provider 服务本身故障或降级（OpenAI 偶发 500 错误）

**根因**：Zify 的核心体验完全依赖外部 API，而外部 API 不受你控制。

**一期是否处理**：✅ 需要。已在 `07-zify-LLM-api-calling.md` 中设计了超时、重试、Provider 切换、熔断方案。额外建议：
- 在模型管理中配置多个 Provider（如 OpenAI + DeepSeek 作为备用），避免单 Provider 依赖
- 提示用户选择较快的模型（GPT-4o-mini vs GPT-4o）以减少等待时间

---

### 2. 文档解析 + 向量化阻塞资源 ⚠️ 严重

**现象**：用户上传文档后，其他人对话变慢或超时。

**触发条件**：
- 同时有 2-3 个用户上传大文档（10MB+ PDF），解析 + 分块 + 调用 Embedding API 生成向量
- 解析是 CPU 密集操作（PDF 解析），向量化是 IO 密集（调 Embedding API），两者叠加占用资源

**根因**：文档处理和用户对话共用同一个 Spring Boot 进程，互相争抢资源。

**一期是否处理**：⚠️ 部分处理。
- ✅ 做的：文档上传后异步处理（虚拟线程），不阻塞 HTTP 请求，前端通过 Redis 轮询进度
- ✅ 做的：限制并发解析数量（例如同时最多 2 个文档在解析，其余排队）
- ❌ 不做的：独立文档处理服务。50 人不会频繁上传文档，排队机制足够

---

### 3. PGVector 检索随数据量增长变慢 ⚠️ 中等

**现象**：知识库检索耗时从毫秒级涨到秒级，拖慢 Agent 对话响应。

**触发条件**：
- 单个知识库 chunk 数超过 10 万条，且没有合适的索引
- 多个知识库同时检索（Agent 绑定了 3+ 个知识库）

**根因**：PGVector 的暴力扫描（无索引）是 O(n) 复杂度，数据量大了线性增长。

**一期是否处理**：✅ 需要，在建表时就做好。
- 为 PGVector 的 Embedding 列创建 HNSW 索引（近似最近邻，查询速度快几个数量级）：

```sql
CREATE INDEX idx_chunk_embedding ON document_chunk
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);
```

- 限制单个知识库的文档数量上限（一期设 100 个文档，chunk 数自然可控）

---

### 4. MySQL 对话消息表膨胀 ⚠️ 中等

**现象**：对话历史查询变慢，打开对话页加载时间变长。

**触发条件**：
- 50 人每天各 10-20 轮对话，半年后消息表达到百万级
- 加载对话历史时 `SELECT * FROM message WHERE conversation_id = ? ORDER BY created_at` 在大表上变慢

**根因**：消息表是系统中增长最快的表，高频写入 + 按时间查询。

**一期是否处理**：✅ 需要，在建表时就做好。
- `conversation_id` + `created_at` 建联合索引
- 对话历史分页加载，不一次查全部
- 一期不做分表分库（数据量远没到），只做索引和分页

---

### 5. SSE 长连接占用资源 ⚠️ 低

**现象**：并发对话用户多时，后端连接数打满。

**触发条件**：
- 50 人中 20 人同时在和 Agent 对话，每人一个 SSE 长连接
- 每个 SSE 连接占一个虚拟线程（虽然很轻量，但连接有内存开销）

**根因**：SSE 连接从建立到结束全程保持（30-120 秒）。

**一期是否处理**：❌ 不需要。
- 50 人同时在线对话的概率极低（内部系统，不是面向公众的）
- 即使 50 人同时对话，虚拟线程 + 8C16G 机器轻松应对
- 如果后续扩展到千人，再做连接池优化和限流

---

### 6. 单点故障 ⚠️ 低

**现象**：Spring Boot 实例挂了，整个系统不可用。

**触发条件**：
- OOM、未捕获异常、LLM 调用超时导致的线程泄漏
- K8s Pod 重启

**根因**：所有组件都是单实例。

**一期是否处理**：⚠️ 部分。
- ✅ 做的：K8s 自带的 Pod 自动重启（`restartPolicy: Always`），挂了自动拉起
- ✅ 做的：Spring Boot Actuator 健康检查端点，K8s liveness/readiness probe 配置
- ❌ 不做的：多副本部署、主从数据库。50 人内部系统，几分钟不可用可以接受

---

## 总结

| 排名 | 瓶颈 | 严重程度 | 一期处理 |
|------|------|---------|---------|
| 1 | LLM API 响应慢 + 限流 | ⚠️ 最严重 | ✅ 超时/重试/熔断/多 Provider |
| 2 | 文档解析抢占资源 | ⚠️ 严重 | ⚠️ 异步 + 并发限制 |
| 3 | PGVector 检索变慢 | ⚠️ 中等 | ✅ 建表时加 HNSW 索引 |
| 4 | 消息表膨胀 | ⚠️ 中等 | ✅ 建表时加索引 + 分页查询 |
| 5 | SSE 长连接 | ⚠️ 低 | ❌ 50 人不需要 |
| 6 | 单点故障 | ⚠️ 低 | ⚠️ K8s 自动重启 + 健康检查 |

**规律**：一期需要处理的都是"建表/编码时就顺手做的"（索引、异步、重试），不需要额外引入中间件或复杂架构。真正需要架构升级的瓶颈（多副本、分库分表、消息队列）在 50 人规模下不会触发。
